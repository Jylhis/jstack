# Validate

You are running the `/validate` workflow. Your job is to stress-test assumptions,
quantify risks, and produce a validation report with a clear go/no-go recommendation.

**HARD GATE:** Do NOT write code or begin implementation. Your only output is
`validation.md`.

---

## Artifact Contract

```
PRODUCES: validation.md
REQUIRES: research.md (at minimum light research must exist)
OPTIONAL:
  - brainstorm.md (grounds validation in the original problem framing)
CONSUMED BY: /specify, /plan, user decision
```

**Pre-flight check:** If `research.md` does not exist, STOP and tell the user:
"Validation requires research first. Run `/research` (even `/research:light`) to
gather evidence, then re-run `/validate`."

## Sub-Roles

### Auto-Detection Logic

```
IF project context involves revenue/pricing/business model/market/customers paying
  -> :business
ELSE
  -> :feature
```

Signals for :business:
- research.md discusses pricing, revenue, unit economics, market size
- brainstorm.md was generated by :product or :business sub-role
- User mentions customers, revenue, monetization, go-to-market

The user can always override: `/validate:business` or `/validate:feature`.

### :business

Focus: Unit economics, market risk, competitive moat, customer acquisition cost,
willingness-to-pay, go/no-go recommendation with financial reasoning.

### :feature

Focus: Technical feasibility, scope risk, effort estimation, dependency risk,
maintenance burden, go/no-go recommendation with engineering reasoning.

## Steps

### Step 1: Load Context

1. Read `research.md`. Extract key findings, recommendation, and unknowns.
2. Read `brainstorm.md` if it exists. Extract the selected direction and assumptions.
3. Read `constitution.md` if it exists. Note constraints.
4. Compile a list of all assumptions (explicit and implicit) from both documents.

### Step 2: Detect Sub-Role

Apply the auto-detection logic above. Announce:
"Running as :business / :feature based on [signal]. Override with `/validate:<role>`."

### Step 3: Enumerate Assumptions

List every assumption the idea rests on. For each assumption:
1. **State it clearly** -- one sentence.
2. **Evidence status** -- supported (cite source), unsupported, or contradicted.
3. **Consequence if wrong** -- what breaks if this assumption fails.

Present the assumption list to the user via AskUserQuestion:
"These are the assumptions I found. Any I missed? Any you want to challenge?"

### Step 4: Risk Analysis

**For :business:**
1. Unit economics estimation (even rough napkin math).
2. Market sizing -- TAM/SAM/SOM with clear methodology.
3. Competitive landscape -- who else is here, what's the moat.
4. Customer acquisition -- how do users find this, what does it cost.
5. Willingness-to-pay signal -- any evidence people would pay, and how much.

**For :feature:**
1. Technical feasibility -- can this be built with known tools and patterns.
2. Scope estimation -- S/M/L/XL with specific reasoning.
3. Dependency risk -- external services, libraries, APIs that could break.
4. Maintenance burden -- ongoing cost of keeping this alive.
5. Migration/rollback risk -- how hard is it to undo if it fails.

**For both:**
- Build a risk matrix: likelihood (low/med/high) x impact (low/med/high).
- Identify the top 3 risks that could kill the project.
- For each top risk, define a mitigation strategy or early warning signal.

### Step 5: Stress-Test via Inversion

Ask the question backwards:
1. "What would have to be true for this to fail spectacularly?"
2. "If a smart skeptic had 5 minutes to argue against this, what would they say?"
3. "What's the most likely reason this gets abandoned halfway through?"

Use these inversions to find blind spots not covered by Step 4.

### Step 6: Go / No-Go Recommendation

Synthesize all evidence into a clear recommendation:

- **GO** -- evidence supports proceeding. State the top risk to watch.
- **GO WITH CONDITIONS** -- proceed, but only if specific conditions are met first. List the conditions.
- **NO-GO** -- evidence does not support proceeding. State what would change the recommendation.
- **INSUFFICIENT DATA** -- cannot make a recommendation. State exactly what evidence is needed.

### Step 7: Write Artifact

Write `validation.md` with the following structure.

## Important Rules

- **Do not proceed without research.md.** This is a hard requirement. Light research is enough.
- **Quantify where possible.** "Risky" is not useful. "3 of 5 assumptions unsupported" is useful.
- **Inversions are mandatory.** Step 5 catches blind spots that forward analysis misses.
- **Be honest about uncertainty.** "Insufficient data" is a valid recommendation. Do not fake confidence.
- **One question at a time.** Never batch multiple questions into one AskUserQuestion.
- **Respect the recommendation.** If the evidence says no-go, say no-go. Do not soften bad news.
