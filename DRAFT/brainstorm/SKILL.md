# Brainstorm

You are running the `/brainstorm` workflow. Your job is to generate divergent ideas,
challenge assumptions, and produce a structured brainstorm document that downstream
skills can consume.

**HARD GATE:** Do NOT write code, create specs, or begin implementation. Your only
output is `brainstorm.md`.

---

## Artifact Contract

```
PRODUCES: brainstorm.md
REQUIRES: nothing (entry point skill)
OPTIONAL:
  - constitution.md (if exists, respect project principles)
  - audit.md (if exists, grounds :scoped sub-role in real architecture)
CONSUMED BY: /research, /validate, /specify
```

---

## Sub-Roles

### Auto-Detection Logic

```
IF no codebase exists AND user description mentions users/customers/UX/personas
  -> :product
ELSE IF user description mentions revenue/market/pricing/business/monetization
  -> :business
ELSE IF codebase exists (audit.md present OR .git with meaningful history)
  -> :scoped
ELSE
  -> :engineer
```

The user can always override with explicit syntax: `/brainstorm:product`, etc.

### :engineer (default)

Focus: Technical solutions, architecture patterns, tool choices, system design.
Thinking style: "What are the technically interesting ways to solve this?"
Artifacts to read: None required.

### :product

Focus: User needs, personas, UX flows, feature prioritization, jobs-to-be-done.
Thinking style: "What would make users love this?"
Artifacts to read: None required.

### :business

Focus: Revenue models, market positioning, pricing, competitive landscape, unit economics.
Thinking style: "How does this become a sustainable business?"
Artifacts to read: None required.

### :scoped

Focus: Improvements within an existing codebase. Refactoring opportunities, feature
additions that fit the current architecture, tech debt reduction strategies.
Thinking style: "Given what we have, what's the highest-leverage change?"
Artifacts to read: audit.md (strongly recommended), constitution.md (if exists).

## Steps

### Step 1: Gather Context

1. Read `constitution.md` if it exists. Note any principles that constrain brainstorming.
2. Read `audit.md` if it exists (especially for :scoped).
3. Read `brainstorm.md` if it exists from a prior session. Note what was already explored.
4. Run `git log --oneline -20 2>/dev/null` to understand recent project activity.
5. Use AskUserQuestion to understand the problem space:
   - "What are you trying to figure out? Give me the raw problem, not a solution."

### Step 2: Detect Sub-Role

Apply the auto-detection logic above. Announce the detected role:
"Running as :engineer / :product / :business / :scoped based on [signal]. Override with `/brainstorm:<role>` if this is wrong."

### Step 3: Divergent Ideation

Generate 5-8 distinct ideas or directions. For each idea:

1. **Name** -- a memorable label.
2. **One-liner** -- what it is in one sentence.
3. **Why it might work** -- the core insight or leverage point.
4. **Why it might fail** -- the biggest risk or assumption.
5. **Effort signal** -- S / M / L / XL gut estimate.

Rules for ideation:
- At least one idea must be contrarian or non-obvious.
- At least one idea must be the "boring" proven approach.
- For :scoped, at least two ideas must reuse existing code/patterns from audit.md.
- Do not self-censor. Include ideas you think might be wrong.

### Step 4: Convergence

1. Group ideas by theme if natural clusters emerge.
2. Identify the 2-3 strongest ideas and explain why.
3. Identify dangerous assumptions shared across multiple ideas.
4. Use AskUserQuestion to present the top ideas and ask:
   "Which of these resonates? Or should we push in a different direction?"

### Step 5: Deepen Selected Direction

Based on user feedback:
1. Expand the selected idea(s) with more detail.
2. Identify open questions that `/research` should investigate.
3. Identify risks that `/validate` should stress-test.
4. Note any constraints from constitution.md that apply.

### Step 6: Write Artifact

Write `brainstorm.md` with the following structure.

## Important Rules

- **Diverge before converging.** Generate breadth first, narrow second.
- **No implementation.** Do not write code, create files, or scaffold projects.
- **One question at a time.** Never batch multiple questions into one AskUserQuestion.
- **Respect the constitution.** If constitution.md exists, note where ideas conflict with stated principles.
- **Name your assumptions.** Every idea rests on assumptions. Make them explicit.
