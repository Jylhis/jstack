# Research

You are running the `/research` workflow. Your job is to gather evidence, analyze
options, and produce a structured research document that downstream skills can consume.

**HARD GATE:** Do NOT write code or begin implementation. Your only output is
`research.md`.

---

## Artifact Contract

```
PRODUCES: research.md
REQUIRES: nothing (can be entry point)
OPTIONAL:
  - brainstorm.md (if exists, focuses research on selected directions and open questions)
CONSUMED BY: /validate, /specify, /plan
```

---

## Sub-Roles

### Auto-Detection Logic

```
IF /validate is likely the next step (business decision, go/no-go, investment)
  -> :deep
ELSE
  -> :light
```

Signals that suggest :deep:
- User mentions "validate", "decision", "go/no-go", "investment", "should we"
- brainstorm.md exists and has a "Risks to Validate" section
- The topic involves money, contracts, migrations, or irreversible choices

The user can always override: `/research:deep` or `/research:light`.

### :light (default)

Scope: Quick survey. 3-5 sources. Enough to make an informed choice.
Depth: Surface-level comparison, key tradeoffs identified, recommendation given.
Time target: Fast -- minutes, not hours.

### :deep

Scope: Thorough investigation. 8-15 sources. Quantitative where possible.
Depth: Detailed analysis, edge cases explored, risk matrix, multiple perspectives.
Time target: Thorough -- take the time needed to be rigorous.

## Steps

### Step 1: Understand the Research Question

1. Read `brainstorm.md` if it exists. Extract:
   - The selected direction and its open questions
   - Risks flagged for investigation
   - Assumptions that need evidence
2. If no brainstorm.md, use AskUserQuestion:
   - "What specific question do you need answered? What decision does this inform?"
3. Restate the research question in one sentence. Confirm with the user if ambiguous.

### Step 2: Detect Sub-Role

Apply the auto-detection logic above. Announce:
"Running as :light / :deep based on [signal]. Override with `/research:<role>`."

### Step 3: Define Research Axes

Before searching, define 3-5 axes of comparison or investigation. Examples:
- Technical feasibility
- Cost / pricing
- Community health / maintenance status
- Performance characteristics
- Migration complexity
- Security posture

Present axes to the user via AskUserQuestion:
"I plan to research along these axes: [list]. Anything to add or change?"

### Step 4: Gather Evidence

**For :light:**
1. Search for 3-5 high-quality sources using WebSearch.
2. Read existing codebase patterns (Grep/Glob) if the question involves the current project.
3. Summarize findings per axis.

**For :deep:**
1. Search for 8-15 sources using WebSearch across multiple query formulations.
2. Read existing codebase patterns if relevant.
3. Look for quantitative data: benchmarks, adoption numbers, issue counts.
4. Identify conflicting viewpoints and note the disagreement.
5. Check recency -- flag sources older than 18 months on fast-moving topics.

### Step 5: Analyze and Synthesize

1. For each axis, summarize what the evidence says.
2. Identify consensus views vs. contested claims.
3. Flag gaps -- what could not be determined from available sources.
4. **For :deep only:** Build a comparison matrix if evaluating alternatives.
5. **For :deep only:** Identify second-order effects and edge cases.

### Step 6: Form Recommendation

1. State a clear recommendation with reasoning.
2. State what evidence would change the recommendation.
3. List remaining unknowns that /validate should stress-test.
4. If the evidence is genuinely inconclusive, say so -- do not force a recommendation.

### Step 7: Write Artifact

Write `research.md` with the following structure.

## Important Rules

- **Evidence over opinion.** Cite sources. Distinguish between established fact, emerging consensus, and speculation.
- **No implementation.** Do not write code, install packages, or run benchmarks. Flag when hands-on testing is needed.
- **Recency matters.** Flag stale sources. Prefer sources from the last 12 months for fast-moving topics.
- **Acknowledge uncertainty.** "I don't know" is a valid finding. Do not pad with filler.
- **One question at a time.** Never batch multiple questions into one AskUserQuestion.
- **Respect scope.** :light stays light. Do not upgrade to :deep unless the user asks.
